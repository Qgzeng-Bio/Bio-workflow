# Install proposal template

`scripts/program_onboard.py plan-install <program>` writes both
`install_proposal.json` and `install_proposal.md`. The JSON file is the execution
contract; the Markdown file is for review before confirmation.

## Required JSON fields

- `program_key`: normalized program key used for reports and draft cards.
- `program_name`: original program name supplied by the user.
- `package_name`: package name proposed for installation.
- `version_spec`: optional version constraint. Empty means latest available from
  the selected source.
- `source_type`: one of `conda`, `container`, `github`, `source`, or `binary`.
- `install_method`: executable install route. Third-version automatic execution
  supports only Conda/Mamba methods.
- `target_env`: target Conda environment name, normally `bio_<program_key>`.
- `channels`: Conda channels, normally `conda-forge`, `bioconda`, `defaults`.
- `commands`: human-readable command strings.
- `command_argv`: structured command argv. `install` rebuilds the expected argv
  from proposal fields and requires an exact match.
- `expected_writes`: paths or environment targets expected to change.
- `network_required`: whether the proposal needs network access.
- `risk_notes`: review notes before confirmation.
- `reuse_existing`: whether the proposal explicitly installs into an existing env.
- `created_at`: timestamp for the proposal.
- `proposal_schema_version`: schema marker used by `install`.
- `created_by`: generator marker used by `install`.
- `evidence_dir`: evidence bundle directory. For `install`, this must match the
  directory containing `install_proposal.json`.

## Conda command contract

Default independent-env proposal:

```bash
mamba create -y -n bio_<program_key> -c conda-forge -c bioconda -c defaults <package>[=<version>]
```

If `mamba` is unavailable when the proposal is created, the command may use
`conda create` instead. If the target env already exists, proposal generation must
block unless `--reuse-existing` is supplied. Reuse proposals install into the
existing env with `mamba install` or `conda install`.

For Conda fallback routes, the review record must also state the solver, channel
priority, known failure risks such as slow solves or missing packages, and the
validation command that will prove the executable works after activation.

## Container proposal contract

When an official Docker/Singularity/Apptainer image exists, prefer a container
proposal before attempting to solve a complex Conda `environment.yml` or
`env.yml`. The proposal must record:

- official image URI and tag or digest when available
- expected `.sif` path or cache target
- expected image/download size when known
- runtime command, bind paths, working directory, and environment variables
- proxy requirement or `none`
- maximum retry count and stop conditions for pull/build attempts
- fallback route if the image is unreachable, too large, or runtime-incompatible
- whether the image is official/maintainer-documented or still untrusted
- why Conda is not the first route, or why Conda is still required

Container proposals are review records only. Do not pull, build, convert, or run
an image until the user confirms a separate execution plan.

## Execution gate

`install` must not run unless all are true:

- the proposal JSON was generated by `program_onboard.py`
- the proposal file is named `install_proposal.json` and lives under a project
  `reports/program-onboarding/<program_key>/<timestamp>/` evidence bundle
- `source_type` is `conda`
- `target_env` is a simple Conda env name, not a path
- `package_name`, `channels`, and `version_spec` contain only safe Conda tokens
- `command_argv` exactly matches the command rebuilt from the proposal fields
- the target env does not already exist, unless the proposal explicitly sets
  `reuse_existing`
- the user runs `install --proposal <json> --yes`

Source, binary, GitHub, and container routes are proposal-only in third-version
onboarding. They require separate review and confirmation before any manual
installation, download, or write under `/data9/home/qgzeng/tools/`.
